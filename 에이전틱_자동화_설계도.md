# 에이전틱 업무 자동화 모델 설계도

> **목적**: 웹·뉴스·보고서·논문·발표 자료를 검색·선별하여 고품질 콘텐츠로 재가공하고,
> 독자 맞춤형으로 카카오워크에 게시하며, 게시물 관련 질문에 자동 응답하는
> **Claude Code 기반 에이전틱 자동화 시스템**의 개발·시현 설계도.
>
> 본 문서는 [context.md](context.md)의 업무 정의와 참조 아키텍처를 바탕으로 작성되었습니다.

## 1. 한눈에 보는 시스템

```
[주제 입력]  /brief <주제> <대상방>
     │
     ▼
┌─────────────────────────────────────────────────────────────────────┐
│  1. researcher (서브에이전트)   웹·뉴스·보고서·논문·발표 검색/수집      │
│       │   · WebSearch / WebFetch / firecrawl                          │
│       │   · 출처 URL · 발행일 · 매체 메타데이터 보존                    │
│       ▼                                                               │
│  2. curator (서브에이전트 + curate 스킬)   신뢰도·최신성·관련성 선별   │
│       │   · 품질 루브릭 점수화 → 상위 N개만 통과                       │
│       ▼                                                               │
│  3. writer (메인 + content-production + audience-fit 스킬)            │
│       │   · 사실기반 종합 · 논리 구조화 · 가독성 · [번호] 인용         │
│       │   · audience/briefs/<방> 카드로 톤·깊이 맞춤                   │
│       ▼                                                               │
│  4. reviewer (서브에이전트)   팩트체크·인용 검수 (품질 게이트)         │
│       │   · 통과 못 하면 3번으로 되돌림 (loop)                         │
│       ▼                                                               │
│  5. kakaowork-post (스킬)   지정된 카카오워크 방에 게시                │
│       │   · 게시본 + 출처를 archive/ 에 보관                          │
│       ▼                                                               │
│  [게시 완료] ──────────────────────────────────────────────┐         │
└────────────────────────────────────────────────────────────│─────────┘
                                                              │
                          ┌───────────────────────────────────▼─────────┐
                          │  6. bot 서버 (3단계, 상시 가동)              │
                          │   · 카카오워크 콜백으로 질문 이벤트 수신     │
                          │   · archive/ 근거로 출처 포함 답변 자동 회신 │
                          └──────────────────────────────────────────────┘
```

## 2. 핵심 설계 원칙

| 원칙                             | 설명                                                                                                         |
| -------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| **단계 = 전담 서브에이전트**     | 검색/선별/검수를 격리된 컨텍스트의 서브에이전트로 분리해 노이즈가 본 대화를 오염시키지 않게 한다.            |
| **규칙·양식·맥락 = 스킬로 고정** | 선별 기준·문체·인용 규칙·독자 카드는 SKILL.md에 고정해 매 실행 동일 품질을 보장한다.                         |
| **생성과 검수의 분리**           | writer가 만든 결과를 reviewer가 비평 → 자기검열 한계를 넘어 품질을 끌어올린다.                               |
| **출처는 1급 시민(first-class)** | 수집 단계부터 URL·발행일·매체를 메타데이터로 보존하여 마지막 인용과 봇 답변까지 추적 가능하게 한다.          |
| **독자-정보 적합성 극대화**      | 독자 원본(profiles)과 요약(briefs)을 분리해, 매 실행 시 가벼운 요약을 먼저 참조한다(progressive disclosure). |
| **명시적 오케스트레이션**        | 자동 위임(라우팅)은 불완전하므로 `/brief` 명령에서 각 단계를 **명시적으로** 호출한다.                        |
| **비밀정보는 전부 환경변수**     | 웹훅 URL·App Key·Bot Secret을 코드/SKILL.md에 하드코딩하지 않는다.                                           |

## 4. 컴포넌트 상세

### 4.1 서브에이전트

> Claude Code 내장 서브에이전트(Explore = 읽기 전용 검색, Plan = 단계 설계,
> general-purpose)와, `.claude/agents/`에 정의하는 커스텀 서브에이전트를 함께 활용한다.

| 에이전트       | 역할                                        | 허용 도구                                                | 격리 이유                                                    |
| -------------- | ------------------------------------------- | -------------------------------------------------------- | ------------------------------------------------------------ |
| **researcher** | 다매체 검색·수집, 출처 메타데이터 보존      | WebSearch, WebFetch, firecrawl_search/scrape (읽기 전용) | 검색 노이즈를 본 대화에서 분리 (내장 Explore 패턴의 웹 버전) |
| **curator**    | 수집 자료를 `curate` 루브릭으로 점수화·선별 | Read, curate 스킬                                        | 대량 후보를 별도 컨텍스트에서 압축                           |
| **reviewer**   | 생성물 팩트체크·인용 검증 (품질 게이트)     | Read, WebFetch (재확인)                                  | "비평가" 분리 → 생성+검수 이원화로 품질↑                     |

- 실행 계획 점검은 내장 **Plan** 서브에이전트를 매 실행마다 사용.
- **주의**: 자동 위임이 불완전해 본 세션이 직접 처리하는 경우가 있으므로 `/brief`에서 각 단계를 명시 호출.

### 4.2 스킬

> 커스텀 스킬은 `SKILL.md`가 든 디렉터리로 만들며, 파일시스템 기반이라 업로드 없이 관련 시 자동 발견된다.
> `description` 필드(최대 200자)가 호출의 핵심이므로 명확히 작성한다.

| 스킬                   | 책임            | 핵심 규칙                                                                            |
| ---------------------- | --------------- | ------------------------------------------------------------------------------------ |
| **curate**             | 후보 자료 선별  | 출처 신뢰도 · 1차 출처 우선 · 최신성 · 주제 적합성 · 중복 제거 → 점수화 후 상위 N    |
| **content-production** | 사실기반 재생산 | 구조: `제목 → 핵심요약 3줄 → 본문 → 📎출처`. 쉬운 표현, 주장마다 `[번호]` 인용       |
| **audience-fit**       | 독자 맞춤       | 대상 방의 `briefs/` 카드를 읽어 톤·깊이·강조점 조정. profiles 갱신 시 brief 재간추림 |
| **kakaowork-post**     | 게시            | 완성본을 지정 방 웹훅으로 전송하는 `scripts/send.py` 번들. 전송 전 archive 저장      |

- **스캐폴딩**: `skill-creator` 스킬로 폴더 구조·SKILL.md 자동 생성.
- **보강**: 다주제 일괄은 `/batch`, 품질 반복 개선은 `/loop`로 보강.

### 4.3 슬래시 명령 — `/brief`

```
/brief <주제> <대상방>
```

`researcher → curator → writer(+audience-fit) → reviewer → kakaowork-post`를 순서대로 **명시적으로** 호출하는 오케스트레이션 진입점. `.claude/commands/brief.md`에 단계별 호출 절차를 기술한다.

## 5. 독자 정보 2폴더 설계 (요구사항 핵심)

독자-정보 적합성을 극대화하기 위해 **원본과 요약을 분리**한다.

| 폴더                 | 내용                                                                                   | 관리 주체                               | 사용 시점                    |
| -------------------- | -------------------------------------------------------------------------------------- | --------------------------------------- | ---------------------------- |
| `audience/profiles/` | 부서·역할, 관심 주제, 의사결정 맥락, 금기사항, 선호 길이/톤 등 **풍부한 원본**         | 사람이 관리·갱신                        | brief 재생성 시 참조         |
| `audience/briefs/`   | 각 독자/방마다 5~10줄짜리 **"독자 카드"** (누구·무엇에 관심·어떤 깊이·어떤 톤·피할 것) | audience-fit 스킬이 profiles에서 간추림 | **매 실행 시** 저렴하게 로드 |

> 원리: 스킬의 progressive disclosure와 동일 — 가벼운 요약(brief)을 먼저, 필요 시 원본(profile)을 참조.
> profiles가 갱신되면 brief를 다시 간추리는 "유지보수" 단계를 둔다.

**독자 카드(brief) 예시**

```markdown
# exec-team 독자 카드

- 누구: 경영진 5인 (대표·CFO·영업총괄 등)
- 관심: 시장 트렌드, 경쟁사 동향, 규제 변화, 매출 영향
- 깊이: 결론 우선·근거 압축 (3줄 요약 중시)
- 톤: 간결·단정, 전문용어 최소화
- 피할 것: 장황한 배경 설명, 검증 안 된 추측
```

## 7. 데이터 흐름 — 단계별 입출력 계약

| 단계           | 입력                   | 출력 (다음 단계로)                                        |
| -------------- | ---------------------- | --------------------------------------------------------- |
| researcher     | 주제 키워드            | 후보 목록 `[{title, url, source, date, excerpt}]`         |
| curator        | 후보 목록              | 선별 목록 `[{...후보, score, reason}]` (상위 N)           |
| writer         | 선별 목록 + 독자 brief | 보고 초안 `{title, summary3, body, citations[]}`          |
| reviewer       | 보고 초안 + 원출처     | `{verdict: pass/fail, issues[]}` → fail이면 writer로 회귀 |
| kakaowork-post | 통과한 보고 + 대상 방  | 게시 결과 + `archive/<날짜>_<주제>.md`                    |
| bot (3단계)    | 멤버 질문 + archive    | 출처 포함 답변                                            |

## 8. 오케스트레이션 & 자동 실행

- **수동/단건**: `/brief <주제> <대상방>` 한 명령으로 전체 파이프라인 실행.
- **정기 실행**: 헤드리스 + cron
  ```bash
  claude -p "/brief 호주 건강기능식품 시장 동향 exec-team"
  ```
  또는 Claude Code 스케줄드 태스크(routine)로 등록.
- **다주제 일괄**: `/batch`, **품질 반복**: `/loop`.

## 9. 개발 3단계 로드맵

### 1단계 — 기본 뼈대 + 작동 테스트

- **산출물**: 리포 구조, researcher(기본), 최소 content-production(출처 포함 기본 양식), kakaowork-post(웹훅), `/brief` 명령.
- **테스트**: 한 주제 → 출처 포함 보고 → 테스트 방에 1회 자동 게시 성공.
- **합격 기준**: 단순 주제 한 건이 사람 개입 없이 방까지 도달.

### 2단계 — 심도 검색·가공·공유 + 독자 적합성

- **산출물**: researcher 강화(다매체), curate 루브릭, curator·reviewer 서브에이전트, content-production 고도화(논리·가독성·엄격 인용), audience profiles+briefs+audience-fit, `/batch`·`/loop`, cron 정기화.
- **합격 기준**: 여러 출처에서 선별된 고품질 보고가 독자 맞춤형으로 정기 자동 게시.

### 3단계 — 카카오워크 질문 답변

- **방식**: 카카오워크 콜백 봇. Bot은 시스템 알람 수신뿐 아니라 멤버와 대화 가능 — 검증된 Bot의 Request/Callback URL로 이벤트가 오면 서버가 응답을 회신하는 구조.
- **흐름**: 봇 서버(공개 HTTPS, `bot/`)가 멤버 질문 이벤트 수신 → `archive/`의 해당 기사+출처를 근거로(필요 시 researcher 재호출) 답변 생성 → Bot API로 회신.
- **주의**: 콜백은 비동기 처리, Bot Secret으로 서명 검증, 답변에도 출처 포함.
- **합격 기준**: 게시된 기사 관련 질문에 출처 포함 답변 자동 회신.

## 10. 시현(데모) 시나리오

> 개발 단계와 무관하게 "보여주기"에 최적화한 흐름. 5~10분 분량.

1. **입력**: 발표자가 `/brief 호주 비타민·건기식 규제 변화 exec-team` 실행.
2. **검색 라이브**: researcher가 뉴스·보고서·논문을 수집하는 로그를 화면에 노출(출처 URL이 쌓이는 모습).
3. **선별**: curator가 점수와 탈락 사유를 표로 출력 → "왜 이 자료만 남았는가"를 설명.
4. **생성**: writer가 `제목 → 3줄 요약 → 본문 → 📎출처` 구조의 보고를 생성, exec-team 톤으로 맞춤.
5. **검수**: reviewer가 인용 1건을 원문과 대조, 불일치 시 자동 수정(품질 게이트 시연).
6. **게시**: kakaowork-post가 지정 테스트 방에 게시 → 카카오워크 화면 전환해 실제 메시지 확인.
7. **(3단계) Q&A**: 발표자가 그 방에 "이 규제는 언제 발효되나요?" 질문 → 봇이 출처 포함 답변 자동 회신.

**데모 체크리스트**

## 11. 보안 · 운영 주의사항

- 비밀정보(웹훅 URL·App Key·Bot Secret)는 **전부 환경변수**. `.env.example`만 커밋, 실제 키는 `.gitignore`.
- SKILL.md에 API 키 등 민감정보 하드코딩 금지.
- **1단계는 "동작"이 목표** — 품질 고도화는 2단계에서.
- **3단계 봇은 cron이 아니라 상시 가동 서버**가 필요(콜백 수신·비동기 응답).
- 소비자용 카카오톡 단톡방은 공식 API 부재 → Q&A는 **카카오워크 안에서만** 구현.
