# 에이전틱 업무 자동화 모델 설계도

> **목적**: 웹·뉴스·보고서·논문·발표 자료를 검색·선별하여 고품질 콘텐츠로 재가공하고,
> 독자 맞춤형으로 **슬랙(Slack)**에 게시하며, 게시물 관련 질문에 자동 응답하는
> **Claude Code 기반 에이전틱 자동화 시스템**의 개발·시현 설계도.
>
> 본 문서는 [context.md](context.md)의 업무 정의와 참조 아키텍처를 바탕으로 작성되었습니다.
>
> **v2 (2026-06-06)** — 설계 인터뷰를 통해 24개 핵심 결정을 확정하고 본문에 반영했습니다.
> **v3 (2026-06-06)** — **게시·Q&A 플랫폼을 카카오워크 → 슬랙(Slack)으로 변경.** 카카오워크가 호주에서 서비스되지 않아,
> 호주 사업 현지에서 쓸 수 있는 슬랙으로 전환했습니다. 슬랙은 스레드·이모지·서명 검증·즉시 ACK가 1급 기능이라
> 기존 결정 대부분과 더 깔끔하게 정합하며, 봇은 **Socket Mode**(공개 HTTPS 불필요)·승인은 **Block Kit 버튼**으로 구현합니다.
> 모든 결정의 단일 목록은 [§0 결정 기록(Decision Log)](#0-결정-기록-decision-log)을 참조하세요.

## 1. 한눈에 보는 시스템

```
[주제 입력]  /brief <주제> <대상방>          ※ 모든 단계 입출력은 runs/<run-id>/ 파일에 기록 (D1)
     │
     ▼
┌─────────────────────────────────────────────────────────────────────┐
│  1. researcher (서브에이전트)   웹·뉴스·보고서·논문·발표 검색/수집      │
│       │   · 하이브리드: 기본 WebSearch/WebFetch + 심화 firecrawl (D11) │
│       │   · 출처 URL · 발행일 · 매체 메타데이터 보존                    │
│       │   · 최근 6개월·15~25건 수집 (D8) / archive 인덱스로 기게시 제외 (D6) │
│       ▼                                                               │
│  2. curator (서브에이전트 + curate 스킬)   신뢰도·최신성·관련성 선별   │
│       │   · 품질 루브릭 점수화 → 상위 5건만 통과 (D8)                  │
│       ▼                                                               │
│  3. writer (메인 + content-production + audience-fit 스킬)            │
│       │   · 한국어 본문 + 원문 출처 병기 (D3) · [번호] 인용            │
│       │   · audience/briefs/<방> 카드로 톤·깊이 맞춤                   │
│       ▼                                                               │
│  4. reviewer (서브에이전트)   핵심 수치·고위험 주장 원문 대조 (D18)     │
│       │   · fail → 3번으로 회귀 (최대 N회, 이후 사람 에스컬레이션 D7)  │
│       ▼                                                               │
│  5. slack-post (스킬)   ┌── 스테이징 채널에 "초안"+승인버튼 게시 (D2)┐  │
│       │   · 초안 + 출처를 archive/ 에 보관                       │     │
│       ▼                                                          ▼     │
│  ┌── 승인 대기 ───────────────────────────────────────────────────┐   │
│  │  지정 승인자(D17)가 [승인]/[반려] Block Kit 버튼 클릭(D5)        │   │
│  │     │ 승인 → 대상 채널에 본 게시 / 반려 → 폐기·수정            │   │
│  └─────┼──────────────────────────────────────────────────────────┘   │
│        ▼                                                               │
│  [본 게시 완료] ───────────────────────────────────────────┐         │
└────────────────────────────────────────────────────────────│─────────┘
                                                              │
                          ┌───────────────────────────────────▼─────────┐
                          │  6. 슬랙 봇 (Socket Mode 상시 프로세스, D13) │
                          │   · 승인 버튼 + Q&A 공용 (Web API, D21)      │
                          │   · 게시물 스레드의 질문을 해당 글에 바인딩(D9)│
                          │   · 즉시 ack() 후 archive/ 근거 답변 비동기 회신(D4)│
                          └──────────────────────────────────────────────┘
```

> **흐름 요약**: `researcher → curator → writer → reviewer → (스테이징 초안) → 버튼 승인 → 본 게시 → (스레드 Q&A)`.
> 1단계 데모(D22)는 승인·Q&A 없이 `slack-post`가 테스트 채널로 바로 게시하는 최소 경로만 증명한다.

## 2. 핵심 설계 원칙

| 원칙                             | 설명                                                                                                         |
| -------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| **단계 = 전담 서브에이전트**     | 검색/선별/검수를 격리된 컨텍스트의 서브에이전트로 분리해 노이즈가 본 대화를 오염시키지 않게 한다.            |
| **규칙·양식·맥락 = 스킬로 고정** | 선별 기준·문체·인용 규칙·독자 카드는 SKILL.md에 고정해 매 실행 동일 품질을 보장한다.                         |
| **생성과 검수의 분리**           | writer가 만든 결과를 reviewer가 비평 → 자기검열 한계를 넘어 품질을 끌어올린다.                               |
| **출처는 1급 시민(first-class)** | 수집 단계부터 URL·발행일·매체를 메타데이터로 보존하여 마지막 인용과 봇 답변까지 추적 가능하게 한다.          |
| **독자-정보 적합성 극대화**      | 독자 원본(profiles)과 요약(briefs)을 분리해, 매 실행 시 가벼운 요약을 먼저 참조한다(progressive disclosure). |
| **명시적 오케스트레이션**        | 자동 위임(라우팅)은 불완전하므로 `/brief` 명령에서 각 단계를 **명시적으로** 호출한다.                        |
| **사람 승인 게이트(D2·D17)**     | 경영진 배포물은 완전 자동 게시하지 않는다. 스테이징 채널 초안 → **지정 승인자**의 Block Kit 버튼 승인 → 본 게시. |
| **실행 = 파일로 기록(D1)**       | 서브에이전트는 텍스트만 반환하므로 단계 입출력을 `runs/<run-id>/` 파일에 남겨 추적·재현·디버깅 가능하게 한다. |
| **비밀정보는 전부 환경변수**     | 슬랙 Webhook URL·Bot Token(xoxb)·App Token(xapp)·승인자 목록을 코드/SKILL.md에 하드코딩하지 않는다.          |

| 에이전트       | 역할                                        | 허용 도구                                                | 격리 이유                                                    |
| -------------- | ------------------------------------------- | -------------------------------------------------------- | ------------------------------------------------------------ |
| **researcher** | 다매체 검색·수집, 출처 메타데이터 보존      | WebSearch, WebFetch, firecrawl_search/scrape (읽기 전용) | 검색 노이즈를 본 대화에서 분리 (내장 Explore 패턴의 웹 버전) |
| **curator**    | 수집 자료를 `curate` 루브릭으로 점수화·선별 | Read, curate 스킬                                        | 대량 후보를 별도 컨텍스트에서 압축                           |
| **reviewer**   | 생성물 팩트체크·인용 검증 (품질 게이트)     | Read, WebFetch (재확인)                                  | "비평가" 분리 → 생성+검수 이원화로 품질↑                     |

- **researcher 검색 전략 (D8·D11)**: 1차는 내장 WebSearch/WebFetch로 최근 6개월 15~25건 수집, 페이월·동적 페이지·핵심 기사만 firecrawl scrape/extract로 보강. 수집 전 `archive/posted-index.md`를 읽어 기게시 URL은 후보에서 제외(D6).
- **reviewer 검증 깊이 (D18)**: 한국어 본문의 **핵심 수치·고유명사·규제 발효일 등 고위험 주장만** 원출처를 재 fetch해 대조하고, 그 외 인용은 URL 존재·접근성만 확인한다. (영어 원문 ↔ 한국어 본문 번역 정확성도 이 고위험 주장에 한해 점검)
- **루프 수렴 (D7)**: reviewer fail 시 writer로 회귀하되 **최대 2~3회**. 이후에도 fail이면 중단하고 미해결 이슈를 표시해 스테이징방/운영자에게 에스컬레이션.
- 실행 계획 점검은 내장 **Plan** 서브에이전트를 매 실행마다 사용.
- **주의**: 자동 위임이 불완전해 본 세션이 직접 처리하는 경우가 있으므로 `/brief`에서 각 단계를 명시 호출.

### 4.2 스킬

> 커스텀 스킬은 `SKILL.md`가 든 디렉터리로 만들며, 파일시스템 기반이라 업로드 없이 관련 시 자동 발견된다.
> `description` 필드(최대 200자)가 호출의 핵심이므로 명확히 작성한다.

| 스킬                   | 책임            | 핵심 규칙                                                                            |
| ---------------------- | --------------- | ------------------------------------------------------------------------------------ |
| **curate**             | 후보 자료 선별  | 출처 신뢰도 · 1차 출처 우선 · 최신성(최근 6개월 가중) · 주제 적합성 · **archive 인덱스 대비 중복 제거(D6)** → 점수화 후 상위 5건(D8) |
| **content-production** | 사실기반 재생산 | 구조: `제목 → 핵심요약 3줄 → 본문 → 📎출처`. **한국어 본문 + 핵심 수치·인용은 원문(영어) 병기(D3)**. 쉬운 표현, 주장마다 `[번호]` 인용. **요약·재서술 원칙, 원문 장문 복제 금지(D12)** |
| **audience-fit**       | 독자 맞춤       | 대상 방의 `briefs/` 카드를 읽어 톤·깊이·강조점 조정. (재간추림은 `/refresh-brief`로 수동 트리거 — D15) |
| **slack-post**         | 게시            | 완성본을 **스테이징 채널에 초안+승인버튼으로** 전송(D2), 전송 전 `archive/` 저장 + `posted-index` 갱신. 전송 채널은 **슬랙 Web API 일원화(D21)** — 1단계 한정 Incoming Webhook 스캐폴드 허용 |

- **스캐폴딩**: `skill-creator` 스킬로 폴더 구조·SKILL.md 자동 생성.
- **보강**: 다주제 일괄은 `/batch`, 품질 반복 개선은 `/loop`로 보강.

### 4.3 슬래시 명령

**`/brief <주제> <대상채널>`** — 오케스트레이션 진입점.
`researcher → curator → writer(+audience-fit) → reviewer → slack-post(스테이징 초안)`를 순서대로 **명시적으로** 호출하고, 모든 단계 입출력을 `runs/<run-id>/`에 기록한다(D1). `.claude/commands/brief.md`에 단계별 호출 절차를 기술한다.

- 옵션: `--depth shallow|medium|deep`로 검색 깊이 조절(기본 medium, D8), `--budget <상한>`로 실행별 예산 상한(D16).
- 정기 실행은 `topics.md`의 주제를 cron이 소비(D14).

**`/refresh-brief <대상채널>`** — `audience/profiles/<채널>`에서 `audience/briefs/<채널>` 독자 카드를 재간추림(D15). profiles 수정 후 운영자가 명시적으로 실행해 brief 낙후를 방지한다.

### 4.4 게시 채널 — 슬랙 Web API 일원화 (D21, v3)

**게시·승인·Q&A를 하나의 슬랙 Web API(`chat.postMessage`)로 일원화**한다. 이유:

- 승인을 "Block Kit 버튼"으로 받고(D5), Q&A를 "스레드 바인딩"(`thread_ts`)으로 처리하며(D9), 답변을 **회신**해야 하므로 → 단방향인 Incoming Webhook만으로는 부족하다. 봇 토큰 기반 Web API가 필요.
- 슬랙은 스레드·버튼·회신이 전부 1급 기능이라 카카오워크 때보다 정합이 깔끔하다(웹훅 vs Bot API 긴장 해소).

> **단, 우선순위는 1단계 종단 데모(D22)**. 1단계는 승인·Q&A 없이 `scripts/send.py`가 Incoming Webhook으로 테스트 채널에 "도달"만 증명하는 스캐폴드로 시작하고, 승인(2단계)·Q&A(3단계)가 붙는 시점에 **Web API로 교체**한다. Webhook 스캐폴드가 사장(dead-end)되지 않도록 send.py 인터페이스를 채널 추상화(`--channel webhook|webapi`)로 설계한다.

### 4.5 게시 · 승인 플로우 (D2·D5·D17·D21, v3)

1. `slack-post`가 완성본을 **스테이징 채널**에 "초안"으로 게시(`chat.postMessage`, Web API)하고 `archive/`에 저장. 초안 메시지에 **[승인]/[반려] Block Kit 버튼**을 부착(D5), 버튼 `value`에 run-id를 담는다.
2. **지정 승인자 화이트리스트(D17)**에 속한 슬랙 사용자가 [승인] 버튼 클릭.
   - 화이트리스트 외 사용자의 클릭은 무시(에페메럴 안내) → 오발·무단 게시 차단.
3. 슬랙 봇(Socket Mode)이 버튼 액션을 수신 → 승인 시 **대상 채널에 본 게시**, 반려 시 폐기 또는 수정 요청. 초안 메시지는 결과를 반영해 갱신.
4. 승인 대기가 타임아웃되면 운영자 채널에 알림(D19).

> 버튼 액션 수신을 위해 **슬랙 봇은 1단계 이후 상시 가동**(D13, Socket Mode)이 필요하다. 즉 "스테이징 승인"을 도입하는 순간부터 3단계 봇 인프라를 앞당겨 재사용한다. **Socket Mode라 공개 HTTPS·도메인은 불필요**하고, 상시 떠 있는 프로세스만 있으면 된다.

## 5. 독자 정보 2폴더 설계 (요구사항 핵심)

독자-정보 적합성을 극대화하기 위해 **원본과 요약을 분리**한다.

| 폴더                 | 내용                                                                                   | 관리 주체                               | 사용 시점                    |
| -------------------- | -------------------------------------------------------------------------------------- | --------------------------------------- | ---------------------------- |
| `audience/profiles/` | 부서·역할, 관심 주제, 의사결정 맥락, 금기사항, 선호 길이/톤 등 **풍부한 원본**         | 사람이 관리·갱신                        | brief 재생성 시 참조         |
| `audience/briefs/`   | 각 독자/방마다 5~10줄짜리 **"독자 카드"** (누구·무엇에 관심·어떤 깊이·어떤 톤·피할 것) | audience-fit 스킬이 profiles에서 간추림 | **매 실행 시** 저렴하게 로드 |

> 원리: 스킬의 progressive disclosure와 동일 — 가벼운 요약(brief)을 먼저, 필요 시 원본(profile)을 참조.
> profiles가 갱신되면 운영자가 **`/refresh-brief <방>`(D15)**로 brief를 다시 간추리는 "유지보수" 단계를 둔다(brief 낙후 방지).
> 초기에는 **exec-team 단일 독자(D20)**로 구축하고, sales-team 등은 종단 완성 후 같은 패턴으로 확장한다.

**독자 카드(brief) 예시**

```markdown
# exec-team 독자 카드

- 누구: 경영진 5인 (대표·CFO·영업총괄 등)
- 관심: 시장 트렌드, 경쟁사 동향, 규제 변화, 매출 영향
- 깊이: 결론 우선·근거 압축 (3줄 요약 중시)
- 톤: 간결·단정, 전문용어 최소화
- 피할 것: 장황한 배경 설명, 검증 안 된 추측
```

## 7. 데이터 흐름 — 단계별 입출력 계약

> **계약 저장 위치 (D1)**: 서브에이전트는 구조화 객체가 아닌 **텍스트**를 반환하므로,
> 각 단계는 결과를 `runs/<run-id>/` 아래 파일로 쓰고 다음 단계는 그 파일을 읽는다.
> `/brief` 오케스트레이터는 각 서브에이전트에게 "결과를 이 경로에 JSON/MD로 기록하라"고 지시하고,
> 반환 텍스트는 가벼운 완료 신호·요약만 메인 컨텍스트로 받는다. `runs/<run-id>/`가 그 실행의 SSOT.

| 단계           | 입력 (읽는 파일)             | 출력 (쓰는 파일)                                           |
| -------------- | ---------------------------- | ---------------------------------------------------------- |
| researcher     | 주제 키워드 + `posted-index` | `researcher.json` `[{title, url, source, date, excerpt}]`  |
| curator        | `researcher.json`            | `curator.json` `[{...후보, score, reason}]` (상위 5)       |
| writer         | `curator.json` + 독자 brief  | `draft.md` `{title, summary3, body(ko), citations[(원문병기)]}` |
| reviewer       | `draft.md` + 원출처          | `review.json` `{verdict, issues[]}` → fail이면 writer 회귀(최대 N회) |
| slack-post     | 통과한 `draft.md` + 스테이징 채널 | 스테이징 게시(+승인버튼) + `archive/<날짜>_<주제>.md` + `posted-index` 갱신 |
| 승인 게이트    | 스테이징 게시 + 버튼 액션    | 승인 시 대상 채널 본 게시 / 반려 시 폐기 (D2·D5·D17)       |
| bot (3단계)    | 멤버 질문(스레드) + archive  | 즉시 ack 후 출처 포함 답변 비동기 회신 (D4·D9)             |

- 실행별 비용·호출 수는 `runs/<run-id>/cost.json`에 기록하고 예산 상한(D16)과 대조한다.

## 8. 오케스트레이션 & 자동 실행 (D14·D16)

실행 주체는 **정기 목록 + 수동 하이브리드**.

- **수동/단건**: 운영자가 `/brief <주제> <대상방>`으로 임시 주제 실행.
- **정기 실행**: `topics.md`의 주제 목록을 헤드리스 + cron이 소비.
  ```bash
  # 예: topics.md 각 줄을 읽어 정기 실행 (또는 Claude Code 스케줄드 태스크/routine)
  claude -p "/brief 호주 건강기능식품 시장 동향 exec-team"
  ```
- **비용 가드(D16)**: `/brief` 1회당 토큰/검색 호출 **예산 상한**을 두고 초과 시 경고·중단, `runs/<run-id>/cost.json`에 기록.
- **다주제 일괄**: `/batch`, **품질 반복**: `/loop`.

> 정기 실행이라도 게시는 **스테이징 초안 → 승인(D2)**을 거치므로, 사람이 자리에 없으면 본 게시는 보류된다.

## 9. 개발 3단계 로드맵

> **현재 우선순위(D22): 1단계 종단 데모 먼저.** 최소 기능으로 "사람 개입 없이 방까지 도달"을 빠르게 증명한 뒤 2·3단계를 증축한다.

### 1단계 — 기본 뼈대 + 작동 테스트 ★ 현재 목표

- **산출물**: 리포 구조(`runs/` 포함), researcher(기본), 최소 content-production(한국어 본문+출처 기본 양식), slack-post(**Incoming Webhook 스캐폴드**, D21), `/brief` 명령.
- **테스트**: 한 주제 → 출처 포함 보고 → 테스트 채널에 1회 자동 게시 성공.
- **합격 기준**: 단순 주제 한 건이 사람 개입 없이 채널까지 도달. (이 단계는 승인·스레드 불필요 → Webhook 허용)
- **주의**: `send.py`를 채널 추상화로 설계해 이후 Web API 교체 시 사장되지 않게 한다.

### 2단계 — 심도 검색·가공·공유 + 독자 적합성 + 승인 게이트

- **산출물**: researcher 강화(다매체 하이브리드, D11), curate 루브릭(중복방지 D6), curator·reviewer 서브에이전트(검증 깊이 D18, 루프 수렴 D7), content-production 고도화(한국어+원문 병기 D3, 저작권 원칙 D12), audience profiles+briefs+`/refresh-brief`(D15), **슬랙 Web API 전환 + Block Kit 승인 플로우(D2·D5·D17)**, `/batch`·`/loop`, cron 정기화(D14), 예산 상한(D16).
- **합격 기준**: 여러 출처에서 선별된 고품질 보고가 독자 맞춤형으로 정기 생성되고, **버튼 승인을 거쳐** 자동 게시.
- **인프라 주의**: 승인 버튼 액션을 받으므로(D5) **슬랙 봇이 이 단계부터 상시 가동**(Socket Mode, 3단계 인프라 앞당김).

### 3단계 — 슬랙 질문 답변

- **방식**: 슬랙 봇(2단계 승인 봇과 동일 프로세스, Socket Mode). 봇이 스레드 메시지·멘션 이벤트를 WebSocket으로 수신 → Web API로 회신.
- **흐름(D4·D9)**: 슬랙 봇(`bot/`)이 **게시물 스레드의**(`thread_ts`) 질문 이벤트 수신 → **즉시 `ack()`**로 3초 타임아웃 회피 → 해당 게시물 `archive/` 근거로(필요 시 researcher 재호출) 답변 생성 → `chat.postMessage`로 스레드에 **비동기 회신**.
- **주의**: 스레드 바인딩으로 대상 글 특정(D9), 토큰 기반 인증(서명 검증은 Socket Mode에선 불필요), 답변에도 출처 포함, 최근 N개월 archive만 활성(D24).
- **합격 기준**: 게시된 기사 관련 질문에 출처 포함 답변 자동 회신.

## 10. 시현(데모) 시나리오

> 개발 단계와 무관하게 "보여주기"에 최적화한 흐름. 5~10분 분량.

1. **입력**: 발표자가 `/brief 호주 비타민·건기식 규제 변화 exec-team` 실행.
2. **검색 라이브**: researcher가 뉴스·보고서·논문을 수집하는 로그를 화면에 노출(출처 URL이 `runs/<run-id>/`에 쌓이는 모습).
3. **선별**: curator가 점수와 탈락 사유(중복·노후 포함)를 표로 출력 → "왜 이 자료만 남았는가"를 설명.
4. **생성**: writer가 `제목 → 3줄 요약 → 본문 → 📎출처` 구조의 **한국어 보고(핵심 수치는 원문 병기)**를 생성, exec-team 톤으로 맞춤.
5. **검수**: reviewer가 핵심 수치 1건을 원문과 대조, 불일치 시 자동 수정(품질 게이트 시연).
6. **게시 → 승인**: slack-post가 **스테이징 채널에 초안+승인버튼** 게시 → 지정 승인자가 [승인] 버튼 클릭 → 대상 채널에 본 게시(슬랙 화면 전환해 실제 메시지 확인).
7. **(3단계) Q&A**: 발표자가 그 게시물 **스레드에** "이 규제는 언제 발효되나요?" 질문 → 봇이 즉시 ack 후 출처 포함 답변 비동기 회신.

> 1단계 데모만 보여줄 때는 6번을 "테스트 채널 자동 게시"로 단순화하고 승인·Q&A는 생략한다(D22).

**데모 체크리스트**

- [ ] 테스트용 슬랙 채널 + **스테이징 채널** + 슬랙 앱(봇) 설치
- [ ] 승인자 화이트리스트(슬랙 user ID) 설정(D17)
- [ ] `.env`에 Webhook URL·Bot Token(xoxb)·App Token(xapp) 설정
- [ ] 시연용 독자 카드(`briefs/exec-team.md`) 작성
- [ ] 봇(Socket Mode) 상시 가동·연결 확인(승인/Q&A 데모 시)
- [ ] 안정적 결과를 위한 사전 리허설 1회

## 11. 보안 · 운영 주의사항

- 비밀정보(슬랙 Webhook URL·Bot Token `xoxb-`·App Token `xapp-`·승인자 화이트리스트)는 **전부 환경변수**. `.env.example`만 커밋, 실제 토큰은 `.gitignore`.
- SKILL.md에 토큰 등 민감정보 하드코딩 금지.
- **저작권·인용(D12)**: 원문 장문 복제 금지, 사실·수치 중심 재서술 + 항상 원문 링크, 최단 인용만 허용. 산출물은 **"사내 참고용"** 명시. 실외부 배포 전 별도 검토.
- **프로필 거버넌스(D23)**: 현 시연·내부 단계는 실명 허용하되, 최소 안전장치로 `audience/profiles/`는 `.gitignore` 권고(개인 성향·금기 등 민감 정보의 VCS 유출 방지). 실운영 전 익명화/접근통제 재정비.
- **실패 처리·알림(D19)**: 검색 0건·슬랙 API 장애·승인 타임아웃·예산 초과 시 **운영자 채널(슬랙)**로 단계·사유·`run-id` 통보. 재시도는 기본 수동.
- **archive 관리(D24)**: 최근 N개월만 Q&A·중복방지 활성 유지, 이전 게시물은 압축 보관.
- **승인 인가(D17)**: 스테이징 채널 승인 버튼은 화이트리스트 사용자만 유효, 그 외 클릭은 에페메럴 안내 후 무시.
- **봇 가동(D13)**: 승인(2단계)·Q&A(3단계)용 **상시 가동 프로세스**(cron 아님)가 필요. **Socket Mode**라 공개 HTTPS·인바운드 포트 불필요(아웃바운드 WebSocket). 슬랙 이벤트는 3초 내 `ack()` 후 비동기 처리.
- 슬랙 토큰 권한(스코프)은 최소화: `chat:write`(게시), `channels:history`/`groups:history`(스레드 질문 수신), 버튼·이벤트용 스코프만 부여.
- (참고) 플랫폼을 카카오워크에서 슬랙으로 전환한 이유는 카카오워크가 호주 미서비스이기 때문(v3).
